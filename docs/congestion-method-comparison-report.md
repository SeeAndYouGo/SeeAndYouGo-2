# 혼잡도 예측 방식 비교 보고서

작성일: 2026-03-28

## 1. 목적

이 문서는 실제 데이터를 기준으로 여러 혼잡도 예측 방식을 직접 비교한 결과를 정리한 보고서다.

이번 비교의 목적은 다음 두 질문에 답하는 것이다.

1. 영업 시작 전, 오늘의 혼잡도를 가장 잘 예측하는 단순 방식은 무엇인가
2. 영업 중, 현재까지 들어온 값을 이용해 5분~수십분 앞 혼잡도를 가장 잘 맞추는 방식은 무엇인가

이번 실험은 복잡한 머신러닝보다 먼저 `단순하지만 운영 가능한 방식`을 검증하는 데 초점을 맞췄다.

## 2. 실험 코드

비교 실험은 다음 파일로 구현했다.

- [CongestionMethodComparison.java](/D:/SeeAndYouGo-2/backend/tools/CongestionMethodComparison.java)

이 코드는 DB에서 직접 데이터를 읽어 다음을 수행한다.

1. `connection`을 5분 버킷으로 정제
2. 같은 식당/같은 5분 슬롯 중복은 마지막 값만 유지
3. `07:30 ~ 19:30` 범위만 사용
4. 슬롯 수가 너무 적은 날짜는 제외
5. 이미 확인한 이상일(`2025-06-28`, `2025-09-29`, `2025-12-22`, `2026-01-13`) 제외
6. `menu`, `menu_dish`, `dish`를 조합해 식사타입별 메인메뉴 signature 생성
7. walk-forward 방식으로 테스트 구간을 순서대로 평가

## 3. 평가 구간

- 테스트 구간: `2026-03-01 ~ 2026-03-27`
- 기준 단위: 식당별 5분 슬롯
- 평가 건수:
  - 일반 식당: 약 3,870 슬롯씩
  - 학생생활관: 396 슬롯

학생생활관은 유효 일수가 적어서 표본 수가 상대적으로 적다.

## 4. 비교한 방식

## 4.1 개장 전 예측 방식

현재 시점의 실측값이 아직 없는 상태에서, 과거 데이터와 메뉴/캘린더만으로 예측하는 방식이다.

### A. `slot_mean`

- 같은 식당의 같은 5분 슬롯 평균
- 요일 구분 없음

### B. `weekday_slot_mean`

- 같은 식당
- 같은 요일
- 같은 5분 슬롯 평균

### C. `recent_8week_weekday_slot_mean`

- 같은 식당
- 같은 요일
- 같은 5분 슬롯
- 최근 8주 데이터만 사용

### D. `weekday_slot_menu_backoff`

- 기본은 `weekday_slot_mean`
- 단, 같은 요일/같은 슬롯/같은 식사타입/같은 메뉴 signature 이력이 3회 이상 있으면 그 평균 사용
- 표본이 부족하면 다시 `weekday_slot_mean`으로 fallback

이 방식은 사용자가 말한 “같은 요일의 데이터 중에서 메인메뉴가 같은 데이터들만 가져오는 방식”을 실제 운영 가능한 형태로 바꾼 것이다.

## 4.2 영업 중 예측 방식

현재까지 들어온 오늘 실측값을 활용하는 방식이다.

### E. `last_value`

- 직전 5분 값을 그대로 다음 값으로 사용

### F. `moving_average_3`

- 최근 3개 슬롯 평균

### G. `weekday_blend`

- `0.7 * 직전값 + 0.3 * 같은 요일 같은 슬롯 평균`

### H. `adaptive_trend_blend`

- `0.5 * 직전값 + 0.3 * 최근 3개 평균 + 0.2 * 최근 8주 같은 요일 같은 슬롯 평균`

## 5. 전체 결과

## 5.1 개장 전 예측

| 방법 | MAE | WAPE | RMSE | 해석 |
|---|---:|---:|---:|---|
| `weekday_slot_menu_backoff` | 30.464 | 0.424 | 54.085 | 최고 성능 |
| `weekday_slot_mean` | 33.121 | 0.461 | 61.295 | 메뉴 없이도 강한 baseline |
| `recent_8week_weekday_slot_mean` | 33.121 | 0.461 | 61.295 | 이번 구간에서는 weekday baseline과 거의 동일 |
| `slot_mean` | 45.115 | 0.628 | 81.418 | 요일 구분이 없어서 가장 약함 |

### 해석

1. `같은 식당 + 같은 요일 + 같은 시간대`만 써도 이미 강한 baseline이다.
2. 메뉴 정보를 무조건 강하게 쓰는 것이 아니라, 표본이 충분할 때만 쓰고 아니면 backoff 하는 방식이 가장 좋았다.
3. 즉, “같은 요일 + 같은 메인메뉴만 보자”는 발상은 맞지만, exact match가 부족한 식당이 많아서 fallback이 반드시 필요하다.

## 5.2 영업 중 예측

| 방법 | MAE | WAPE | RMSE | 해석 |
|---|---:|---:|---:|---|
| `last_value` | 4.574 | 0.064 | 10.173 | 최고 성능 |
| `moving_average_3` | 6.834 | 0.095 | 15.892 | 직전값보다 느림 |
| `adaptive_trend_blend` | 8.093 | 0.113 | 16.370 | 복잡하지만 직전값보다 못함 |
| `weekday_blend` | 10.635 | 0.148 | 20.300 | 역사 평균을 섞을수록 오히려 늦어짐 |

### 해석

1. 영업 중에는 역사 평균보다 `지금 막 들어온 값`이 훨씬 강한 신호다.
2. 현재 패턴이 연속적이기 때문에, 단순한 `직전값 유지`가 가장 잘 맞았다.
3. 즉, 실시간 구간에서는 복잡한 규칙보다 `최근 실측값 기반 nowcasting`이 우선이다.

## 6. 식당별 관찰

## 6.1 개장 전 예측

### 제1학생회관

- `weekday_slot_menu_backoff`가 가장 큰 개선을 냈다
- MAE가 `84.317 -> 70.959`로 감소했다
- 반복 메뉴 구조가 많아 메뉴 exact match가 실제로 도움이 된다

### 제2학생회관

- 메뉴 backoff가 소폭 개선했다
- MAE가 `36.898 -> 36.700`
- 메뉴 정보는 있으면 좋지만, 제1학생회관만큼 결정적이지는 않다

### 상록회관, 제3학생회관

- 메뉴 backoff와 weekday baseline이 거의 같았다
- 메뉴 exact match 표본이 부족해서 fallback이 대부분 동작한 것으로 보는 게 맞다

### 학생생활관

- 전체적으로 오차가 크고 표본도 적다
- 별도 모델 또는 별도 룰이 필요하다

## 6.2 영업 중 예측

모든 식당에서 `last_value`가 1위였다.

즉, 영업 중에는 식당별 패턴 차이보다도 `지금 직전에 얼마였는가`가 가장 중요했다.

## 7. “같은 요일 + 같은 메인메뉴” 전략에 대한 실험 결론

이번 실험으로 보면 이 전략은 다음처럼 정리할 수 있다.

### 맞는 부분

- 개장 전 예측에서 메뉴 정보는 분명 도움이 된다
- 특히 반복 메뉴가 많은 식당에서는 효과가 크다

### 틀린 부분

- 모든 식당에서 exact match를 강하게 걸면 표본이 너무 줄어든다
- exact match만 믿으면 오히려 안정성이 떨어진다

### 최종 판단

`같은 요일 + 같은 메인메뉴`는 “주 방식”이 아니라 “보조 방식”으로 써야 한다.

가장 좋은 구조는 다음과 같다.

1. 기본: 같은 식당 + 같은 요일 + 같은 시간대
2. 보정: 같은 메뉴 signature 이력이 충분하면 그 평균으로 보정
3. 부족하면 다시 기본 baseline으로 fallback

즉, 메뉴 exact match는 `조건부 가산점`이지 `기본 필터`가 아니다.

## 8. 지금 기준으로 가장 추천하는 운영 방식

## 8.1 개장 전

`weekday_slot_menu_backoff`

이 방식을 추천하는 이유:

- 구현이 단순하다
- 설명 가능성이 높다
- 메뉴가 유효할 때만 사용해서 과적합 위험이 적다
- 실제 테스트에서 가장 좋았다

실제 운영 규칙:

1. 같은 식당
2. 같은 요일
3. 같은 5분 슬롯
4. 같은 식사타입
5. 같은 메뉴 signature 이력이 3회 이상이면 그 평균
6. 아니면 weekday baseline

## 8.2 영업 중

`last_value`

이 방식을 추천하는 이유:

- 가장 정확했다
- 구현이 가장 쉽다
- API/배치 부담이 작다
- 유지보수도 쉽다

실제 운영 규칙:

1. 직전 5분 값이 있으면 그대로 다음 슬롯 예측
2. 직전값이 없으면 개장 전 baseline 사용

## 8.3 학생생활관

학생생활관은 별도로 다루는 것이 좋다.

권장안:

- v1에서는 별도 규칙 사용
- 또는 혼잡도 레벨만 단순화
- 데이터가 더 쌓이면 별도 모델 학습

## 9. 모델 없이도 충분한가

현재 실험 기준으로는 `모델 없이도 충분히 쓸 만한 1차 버전`을 만들 수 있다.

특히:

- 개장 전: `weekday_slot_menu_backoff`
- 영업 중: `last_value`

이 두 가지만 조합해도 꽤 강한 운영형 시스템이 된다.

즉, 당장 머신러닝 모델을 넣지 않아도 된다.

## 10. 그럼 모델은 언제 필요한가

다음 조건이 생기면 모델 도입 가치가 커진다.

1. 30분, 60분 ahead 예측 정확도를 더 올리고 싶을 때
2. 메뉴, 휴일, 학기 패턴을 더 정교하게 반영하고 싶을 때
3. 학생생활관 같은 특이 패턴 식당을 더 잘 맞추고 싶을 때
4. 날씨, 행사, 시험기간 같은 외부 데이터까지 붙일 때

그때는 `LightGBM`이나 `CatBoost`를 추가하면 된다.

## 11. 최종 결론

이번 비교 실험으로 얻은 결론은 명확하다.

### 결론 1

개장 전 예측은 `같은 식당 + 같은 요일 + 같은 시간대`가 중심이다.

### 결론 2

메뉴 정보는 도움이 되지만, exact match를 강하게 거는 것이 아니라 `표본이 충분할 때만 쓰는 backoff 방식`이 가장 좋다.

### 결론 3

영업 중 예측은 역사 평균보다 `직전 5분 값`이 훨씬 강하다.

### 결론 4

현재 단계에서는 복잡한 모델보다:

- 개장 전 `weekday_slot_menu_backoff`
- 영업 중 `last_value`

조합이 가장 현실적이고 효율적이다.
