# 혼잡도 예측 분석 보고서

작성일: 2026-03-28

## 1. 목적

이 문서는 `SeeAndYouGo` 프로젝트에서 식당 혼잡도를 어떻게 예측하는 것이 가장 현실적인지 정리한 보고서다.

핵심 목표는 다음 두 가지다.

1. 오늘의 특정 시각 혼잡도를 사전에 예측한다.
2. 영업 중에는 현재까지 들어온 실측 데이터를 반영해 5분~60분 앞 혼잡도를 갱신한다.

예측의 대상은 최종적으로 `혼잡도 레벨`일 수 있지만, 실제 모델은 먼저 `연결 수(connected)`를 예측한 뒤 이를 혼잡도 구간으로 변환하는 방식이 더 적합하다.

## 4. 데이터 패턴에서 확인한 점

### 4.1 시간대 패턴

모든 식당에서 점심 피크가 강하게 나타난다.

- 상록회관: 최고 평균 시각 `12:05`
- 생활과학대: 최고 평균 시각 `12:05`
- 제1학생회관: 최고 평균 시각 `12:15`
- 제2학생회관: 최고 평균 시각 `12:05`
- 제3학생회관: 최고 평균 시각 `12:15`
- 학생생활관: 최고 평균 시각 `12:10`

단, 학생생활관은 일반 식당과 다르게 저녁 수요도 높다.

- 학생생활관 점심 평균: 101.6
- 학생생활관 저녁 평균: 103.2

즉, 학생생활관은 별도 모델 또는 최소한 별도 파라미터가 필요하다.

### 4.2 요일 패턴

요일 효과가 매우 크다.

- 대부분 식당은 수요일, 목요일이 강함
- 금요일부터 하락
- 주말은 평일보다 크게 낮음

즉, `같은 시간대`만 보는 것은 부족하고 반드시 `같은 요일`을 함께 봐야 한다.

### 4.3 계절/학기 패턴

월별 평균 connected를 보면 방학과 학기중 차이가 뚜렷하다.

- `2025-01 ~ 2025-02`: 매우 낮음
- `2025-04 ~ 2025-06`: 높음
- `2025-09 ~ 2025-11`: 다시 높음
- `2026-01 ~ 2026-02`: 낮음
- `2026-03`: 다시 상승

즉, `학기중 vs 방학`, `학기 초/중/말`, `시험기간 근처` 같은 시즌 피처가 중요하다.

## 5. 데이터 품질 이슈

예측 전에 정제가 반드시 필요하다.

### 5.1 중복 데이터

특정 날짜와 시각에 동일 레코드가 비정상적으로 많이 반복된다.

대표 사례:

- `2026-01-13 19:25:16`: 식당별 4,374회 반복
- `2025-12-22 19:25:15`: 식당별 2,068회 반복

이런 날짜를 그대로 학습시키면 모델이 실제 패턴이 아닌 수집 오류를 학습하게 된다.

### 5.2 부분 수집일

정상적인 영업일이면 `07:30 ~ 19:30` 기준 145개 5분 슬롯이 거의 채워져야 한다.

일반 식당은 대체로 충분한 일수가 있지만, 일부 날짜는 관측이 매우 적다.

식당별 `140 슬롯 이상` 정상일 수:

- 상록회관: 363일
- 생활과학대: 309일
- 제1학생회관: 363일
- 제2학생회관: 361일
- 제3학생회관: 308일
- 학생생활관: 58일

학생생활관은 부분 수집일 비중이 높아 별도 처리 없이는 바로 학습하기 어렵다.

### 5.3 시간 컬럼 타입

`connection.time`이 문자열이다.

예측 파이프라인에서는 반드시 다음으로 변환해야 한다.

- `timestamp`
- `date`
- `time`
- `5분 bucket`
- `meal type`

## 6. 어떤 정보를 토대로 예측하는 것이 좋은가

예측에 사용할 정보는 중요도 기준으로 다음 순서가 적합하다.

### 6.1 가장 중요한 정보

1. 식당
2. 날짜
3. 5분 시간대
4. 요일
5. 최근 5분, 10분, 15분, 30분, 60분 실측값
6. 전일 같은 시각 값
7. 전주 같은 요일 같은 시각 값
8. 공휴일 여부
9. 학기중/방학 여부
10. 아침/점심/저녁 구분

이 정보들만으로도 강한 baseline 모델이 가능하다.

### 6.2 있으면 좋은 메뉴 정보

메뉴는 유용하지만, 식당마다 활용 방식이 달라야 한다.

우선순위는 다음과 같다.

1. `menu_type` 일치 여부
2. `is_open`
3. 해당 시간대 오픈된 메뉴 수
4. 메인메뉴 이름
5. 메인메뉴 카테고리
6. 평균 가격
7. 평균 평점

메뉴는 단독 기준이 아니라, 시간/요일/계절 정보 위에 얹는 보정 신호로 쓰는 것이 맞다.

### 6.3 현재 DB만으로는 약한 정보

- 리뷰 수
- 좋아요 수
- visitor_count

현재 규모가 작고 시간축 결합이 약해서 주 모델의 핵심 피처로 쓰기에는 부족하다.

## 7. “같은 요일 + 같은 메인메뉴” 전략은 얼마나 유효한가

이 아이디어는 방향은 좋지만, 모든 식당에 동일하게 적용하면 안 된다.

### 7.1 결론

- 제1학생회관: 매우 유효
- 제2학생회관, 제3학생회관: 보조적으로 유효
- 상록회관, 생활과학대: exact match 표본이 적어서 보조 신호 수준
- 학생생활관: breakfast 일부만 유효, 나머지는 sparsity가 큼

### 7.2 근거

`식당 + 식사타입 + 요일 + 메인메뉴` 단위로 과거 반복 수를 봤을 때:

| 식당 | 식사타입 | 평균 반복 횟수 |
|---|---|---:|
| 상록회관 | LUNCH | 1.66 |
| 생활과학대 | LUNCH | 2.00 |
| 제1학생회관 | LUNCH | 35.73 |
| 제2학생회관 | BREAKFAST | 2.79 |
| 제2학생회관 | LUNCH | 2.74 |
| 제3학생회관 | LUNCH | 2.68 |
| 학생생활관 | BREAKFAST | 3.52 |
| 학생생활관 | LUNCH | 1.70 |
| 학생생활관 | DINNER | 1.47 |

즉, 대부분 식당은 `같은 요일 + 같은 메인메뉴`로 exact match를 걸면 과거 표본이 1~3개 수준으로 줄어든다.

이 경우 표본이 너무 적어서 예측이 불안정해진다.

반면 제1학생회관은 반복 메뉴가 많아서 exact match가 강한 신호가 된다.

### 7.3 추천 해석

따라서 `같은 요일 + 같은 메인메뉴`는 다음처럼 써야 한다.

- 기본 필터로 쓰지 않는다
- feature로 넣는다
- 표본이 충분한 식당에서만 backoff rule로 쓴다

권장 규칙:

1. 같은 식당 + 같은 5분 슬롯 + 같은 요일을 기본 집합으로 잡는다.
2. 그 안에서 같은 식사타입을 우선한다.
3. 같은 메인메뉴 이력이 `8회 이상`이면 별도 보정값으로 사용한다.
4. `8회 미만`이면 메인메뉴 exact match 대신 메인메뉴 카테고리 또는 메뉴 수/가격대만 사용한다.

## 8. 추천 예측 방식

## 8.1 최종 권장 방식

가장 현실적인 구조는 다음과 같다.

1. `connection`을 5분 단위로 정제한 학습 테이블을 만든다.
2. 먼저 `connected` 회귀 모델을 학습한다.
3. 예측된 connected를 식당별 분포 기준으로 `여유/보통/혼잡`으로 변환한다.

이 방식이 좋은 이유:

- count 예측이 정보 손실이 적다
- 혼잡도 레벨 구간은 운영 중 조정이 가능하다
- 식당별 임계치를 나중에 쉽게 바꿀 수 있다

### 8.2 모델 선택

1차 모델로는 `LightGBM` 또는 `CatBoost` 회귀를 추천한다.

이유:

- 표 형식 데이터에 강함
- 시간/요일/카테고리 피처를 잘 처리함
- 데이터 규모가 딥러닝을 꼭 써야 할 수준은 아님
- 이상치가 있어도 상대적으로 안정적임

추천 구성:

- 일반 식당용 글로벌 모델 1개
- 학생생활관 전용 모델 1개

## 11. 메인메뉴 정보는 어떻게 쓰는 것이 좋은가

메인메뉴는 다음 3단계로 쓰는 것이 좋다.

### 11.1 1단계: exact match

조건:

- 같은 식당
- 같은 식사타입
- 같은 요일
- 같은 메인메뉴
- 과거 표본 수가 충분함

이 경우:

- 과거 평균
- 과거 분산
- 평균보다 높은 메뉴인지 여부

를 feature로 사용할 수 있다.

### 11.2 2단계: menu family match

exact match 표본이 부족하면 메인메뉴 문자열 그대로 쓰지 말고 메뉴군으로 묶는다.

예:

- 돈까스류
- 덮밥류
- 국밥/찌개류
- 면류
- 볶음밥류
- 샐러드/경식류

이렇게 하면 sparsity를 줄일 수 있다.

### 11.3 3단계: menu summary

메뉴 이름 자체가 불안정하면 다음 집계값만 써도 충분하다.

- 메인메뉴 개수
- 오픈 여부
- 가격대
- 한식/양식/면/분식 등 dept 성격

## 12. 혼잡도 레벨은 어떻게 정할 것인가

식당별 connected 분포 차이가 커서 절대값 기준은 부정확할 수 있다.

따라서 다음 방식이 더 적절하다.

1. 식당별
2. 식사타입별
3. 시간대별

분포를 기준으로 percentile 구간을 만든다.

예시:

- 하위 40%: 여유
- 40% ~ 75%: 보통
- 상위 25%: 혼잡

이 방식은 제1학생회관과 상록회관을 같은 숫자 잣대로 비교하지 않아도 된다는 장점이 있다.

## 14. 최종 결론

현재 DB만으로도 혼잡도 예측은 충분히 가능하다.

다만 핵심은 다음이다.

1. `connection` 원천 데이터를 5분 단위로 정제해서 쓰는 것
2. `같은 요일 + 같은 시간대 + 최근 추세`를 예측의 중심으로 두는 것
3. `메인메뉴 exact match`는 보조 신호로 쓰는 것
4. 학생생활관은 별도 모델 또는 별도 규칙으로 다루는 것
5. 먼저 connected를 예측하고, 그 다음 혼잡도 레벨로 변환하는 것

정리하면 가장 좋은 방향은 다음 한 줄로 표현할 수 있다.

`오늘 혼잡도는 같은 식당의 같은 시간대 패턴을 기본으로 하고, 요일/학기/휴일/오늘 메뉴/최근 실측값으로 보정하는 방식이 가장 안정적이다.`
