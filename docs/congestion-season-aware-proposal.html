1. 먼저 알아둘 용어

MAE

MAE는 예측값과 실제값이 평균적으로 몇 명 차이 나는지를 뜻합니다. 숫자가 작을수록 실제값에 더 가깝습니다.

예를 들어 MAE 10이면, 예측이 실제와 평균 10명 정도 차이 난다는 뜻입니다.

평소값

평소값은 같은 식당 + 같은 요일 + 같은 시간대의 평균 인원입니다. 이번 제안에서는 여기에 같은 시즌을 먼저 붙이는 방식을 기본으로 봤습니다.

예를 들어 2026년 3월 수요일 12시 30분 제1학생회관을 예측할 때는, 2025년 3~6월 수요일 12시 30분 제1학생회관 데이터를 우선 참고합니다.

평소값은 단순한 전체 평균이 아닙니다. 예측하려는 날짜와 비슷한 조건의 과거 데이터를 모아 만든 기준값입니다.

2. 왜 시즌성을 넣어야 하는가

학생식당은 학기 중과 방학의 이용 패턴이 다릅니다. 따라서 모든 과거 데이터를 한 번에 섞어 평균을 내면, 실제보다 높거나 낮게 잡히는 경우가 생깁니다.

3. 이번 문서의 운영 기준

정확도만 최우선으로 보면 식당별 예외를 더 많이 둘 수 있습니다. 다만 운영 단계에서는 규칙이 너무 많아지면 설명과 유지보수가 어려워집니다. 따라서 이번 문서에서는 운영 통일성을 우선해 아래 기준으로 정리했습니다.

4. 시간대별 최종 적용 방향

예측 범위 먼저 보는 값 추가로 보는 값 적용 방향
30분 후 30분 뒤 평소값 지금이 평소보다 얼마나 많은지 또는 적은지 목표 시점의 평소값을 기본으로 두고, 현재 차이는 일부만 가져갑니다.
60분 후 60분 뒤 평소값 현재 차이, 피크 이후 여부 30분 후보다 더 강하게 평소값 쪽으로 되돌아가게 봅니다.
90분 후 90분 뒤 평소값 현재 차이, 피크 이후 여부 더 먼 미래이므로 현재 상태보다 평소 흐름 복귀를 더 강하게 보는 편이 적절했습니다. 전체 MAE는 18.5에서 18.0으로 소폭 개선됐습니다.

30분 후

  • 목표 시점의 평소값을 먼저 봅니다.
  • 현재가 평소보다 많거나 적으면 그 차이를 일부 반영합니다.
  • 현재 상황이 아직 영향을 줄 가능성이 있어 60분 후보다 현재 차이를 더 크게 반영합니다.

60분 · 90분 후

  • 시간이 멀어질수록 현재값만 끌고 가면 늦게 반응하는 문제가 생깁니다.
  • 따라서 목표 시점의 평소값을 먼저 봐야 합니다.
  • 60분 후는 현재 차이를 중간 정도만 반영합니다.
  • 90분 후는 현재 상태보다 평소 흐름으로 돌아가는 쪽을 더 강하게 봅니다.

4.1. 60분 후 예측값 산출 예시

60분 후 예측은 “60분 뒤 평소값”에 “현재가 평소보다 얼마나 다른지”를 일부 더하는 방식입니다.

60분 후 예측값 = 60분 뒤 평소값 + (현재값 - 현재 시점 평소값) × 현재 차이 반영률
항목 예시 값 의미
현재값 80명 지금 식당에 있는 인원입니다.
현재 시점 평소값 60명 같은 식당 · 같은 요일 · 같은 시간 · 같은 시즌의 평균입니다.
60분 뒤 평소값 45명 예측하려는 시각의 과거 평균입니다.
현재 차이 20명 현재값 80명 - 현재 평소값 60명입니다.
현재 차이 반영률 0.4 현재의 특이 상황을 40%만 60분 뒤에 반영합니다.
최종 예측값 53명 45 + 20 × 0.4 = 53명입니다.

위 숫자는 설명용 예시입니다. 실제 반영률은 식당과 예측 범위별 검증 결과를 기준으로 조정합니다.

4.2. 90분 후 혼잡도 예측 방법

90분 후 혼잡도 예측은 현재 시점으로부터 먼 미래를 예측하는 것이므로, 현재의 일시적인 변화보다는 장기적인 '평소 흐름'으로의 복귀에 강하게 중점을 둡니다.

  • 주요 기준: 90분 뒤의 '평소값'을 가장 중요한 기준으로 삼습니다. '평소값'은 같은 식당, 같은 요일, 같은 시간대, 그리고 같은 시즌의 과거 데이터를 기반으로 계산됩니다.
  • 시즌성 반영: 학생식당의 이용 패턴은 학기 중과 방학에 따라 크게 달라지므로, 예측의 정확도를 높이기 위해 시즌을 구분하여 평소값을 산정합니다. (예: 1-2월 겨울 비수기, 3-6월 1학기, 7-8월 여름 비수기, 9-12월 2학기)
  • 보조적 고려 사항: 현재의 혼잡도 차이와 피크 시간 이후 여부도 보조적으로 고려하지만, 예측 시점이 멀기 때문에 이 요소들의 영향력은 상대적으로 약하게 반영됩니다.
  • 예측 방향: 현재 상태의 급격한 변화를 그대로 반영하기보다는, 목표 시점의 평소값으로 천천히 되돌아가는 경향을 더 강하게 예측합니다. 이는 먼 미래 예측의 안정성을 높이는 데 기여하며, 실제 검증 결과 전체 MAE가 18.5에서 18.0으로 소폭 개선되었습니다.

5. 후보별 채택 여부

후보 채택 여부 이유

같은 시즌 · 같은 요일 · 같은 시간대 평균 채택 특히 30분·60분 후에서 평균값 자체가 더 좋아졌습니다.

피크 이후 감소 허용 부분 채택 30분·60분 직접 감소는 과했으므로, 평소값 복귀 방식 안에서만 보조적으로 봅니다.
같은 메인메뉴 기준 비채택 적용 가능한 구간이 약 7% 수준으로 적었습니다.

8. 최종 정리

가장 단순하고 설명하기 쉬운 방향은 다음과 같습니다. 평균값은 학생생활관을 제외하고 시즌성을 반영해서 계산하고, 30분·60분·90분 후는 목표 시점의 평소값을 먼저 본 뒤 현재 차이를 일부만 가져가는 방식입니다. 상록회관과 제3학생회관은 시즌성 적용 시 손해가 있으므로, 통일 기준으로 적용하되 모니터링 식당으로 관리하는 것이 좋습니다.

참고 메모: 복날처럼 메뉴가 거의 고정되는 특수일은 이후 별도 이벤트 플래그나 예외 규칙으로 추가 검토할 수 있습니다.

9. 식당별 예측 공식 요약

각 식당의 혼잡도 예측은 '평균값 기준'과 '예측 범위'에 따라 다음과 같은 규칙을 따릅니다. '평소값'은 같은 식당, 같은 요일, 같은 시간대, 그리고 같은 시즌(학생생활관 제외)의 과거 데이터를 기반으로 합니다.

공통 정의

  • 평소값: 같은 식당, 같은 요일, 같은 시간대, 그리고 같은 시즌(학생생활관 제외)의 과거 데이터를 기반으로 계산된 평균 인원.
  • 현재 흐름 중심: 현재의 혼잡도 값을 기본으로 보되, 최근 변화량(예: 10%)이나 피크 이후 감소 허용 등의 보정을 적용.
  • 현재 차이 반영: 목표 시점의 평소값에 현재 시점의 혼잡도와 평소값 간의 차이(현재 편차)를 더하여 예측값을 보정.

운영 파라미터

파라미터 권장 초기값 설명
시즌 구간 1~2월, 3~6월, 7~8월, 9~12월 평소값을 계산할 때 과거 데이터를 나누는 기준입니다.
30분 현재 차이 반영률 크게 반영 현재 상황이 아직 이어질 가능성을 비교적 크게 봅니다.
60분 현재 차이 반영률 중간 반영 현재 상황과 평소 흐름을 절충합니다.
90분 현재 차이 반영률 약하게 반영 또는 평소값 중심 시간이 멀어질수록 현재 상황보다 평소 흐름을 더 중요하게 봅니다.
학생생활관 평균 기준 전 기간 평균 검증 결과상 시즌성 이득이 크지 않아 단순 기준을 유지합니다.

개념적 적용 (반영률)

  • 크게 반영: 현재 편차의 영향력을 높게 가져가, 현재 상황 변화에 민감하게 반응합니다.
  • 중간 반영: 현재 편차의 영향력을 중간 수준으로 가져가, 평소값과 현재 상황 변화를 균형 있게 고려합니다.
  • 약하게 반영 / 더 약하게 반영: 현재 편차의 영향력을 낮게 가져가, 평소값으로의 회귀 경향을 강하게 유지합니다.
  • 중심: 현재 편차의 영향력을 거의 가져가지 않고, 평소값에 거의 전적으로 의존합니다.

상록회관, 제1학생회관, 제2학생회관, 제3학생회관 (시즌성 평균 적용)

  • 평균값 기준: 시즌성 평균
  • 30분 후 예측: 시즌 평소값 + 현재 차이 크게 반영
  • 60분 후 예측: 시즌 평소값 + 현재 차이 중간 반영
  • 90분 후 예측: 시즌 평소값 + 현재 차이 약하게 반영

생활과학대 (시즌성 평균 적용)

  • 평균값 기준: 시즌성 평균
  • 30분 후 예측: 시즌 평소값 중심
  • 60분 후 예측: 시즌 평소값 중심
  • 90분 후 예측: 시즌 평소값 중심

학생생활관 (전 기간 평균 적용)

  • 평균값 기준: 전 기간 평균
  • 30분 후 예측: 전 기간 평소값 + 현재 차이 약하게 반영
  • 60분 후 예측: 전 기간 평소값 + 현재 차이 더 약하게 반영
  • 90분 후 예측: 전 기간 평소값 중심

10. DB 연동 프로젝트 제안

정적 HTML에 DB 접속 정보를 직접 넣는 방식은 권장하지 않습니다. 브라우저에 DB 주소와 계정이 노출되기 때문입니다. 실제 프로젝트에서는 백엔드가 DB를 조회하고, 프론트엔드는 백엔드 API만 호출하는 구조가 안전합니다.

구성 역할 초기 구현 범위
DB 실제 혼잡도, 식당, 메뉴, 시간대 데이터 저장 기존 MySQL 사용
백엔드 API 평소값 계산, 30·60·90분 예측값 계산, 결과 반환 /api/congestion/predictions 형태로 시작
프론트엔드 식당과 날짜 선택, 그래프 표시, 파라미터 표시 현재 HTML 화면을 API 기반 화면으로 전환
우선은 “식당 + 기준 시각”을 입력하면 현재값, 현재 평소값, 30분 뒤 평소값, 60분 뒤 평소값, 90분 뒤 평소값, 각 예측값, 적용 파라미터를 함께 반환하는 API부터 만드는 것이 좋습니다.
