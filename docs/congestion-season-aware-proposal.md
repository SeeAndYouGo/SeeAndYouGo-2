# 학생식당 혼잡도 예측 적용 제안서

- 작성일: `2026-04-05`
- 규칙 기준 데이터: `2025-01-01 ~ 2025-12-31`
- 검증 데이터: `2026-01-01 ~ 2026-03-27`
- 목적: 제3자가 바로 이해할 수 있도록, 학생식당 혼잡도 예측을 서비스에 어떻게 적용하면 좋은지 쉽고 짧게 정리

## 이번 문서의 운영 원칙

- `학생생활관 제외 전 식당`: 시즌성 평균 사용
- `학생생활관`: 전 기간 평균 유지
- 이유: 운영 단순성과 설명 편의성을 우선하면서도, 전체 기준으로는 시즌성 반영 효과가 확인됐기 때문

## 먼저 알아둘 용어

### MAE
- `MAE`는 예측값과 실제값이 평균적으로 몇 명 차이 나는지 뜻합니다.
- 숫자가 작을수록 실제값과 가깝습니다.

### 평소값
- `평소값`은 같은 식당, 같은 요일, 같은 시간대의 평균 인원입니다.
- 이번 제안에서는 여기에 `같은 시즌`을 먼저 붙이는 방식을 기본으로 봤습니다.
- 예: `2026-03-18` 수요일 `12:30` 제1학생회관 예측 시, `2025년 3~6월` 수요일 `12:30` 제1학생회관 데이터를 우선 참고

## 왜 시즌성을 넣어야 하는가

학생식당은 학기 중과 방학의 이용 패턴이 다릅니다. 따라서 모든 과거 데이터를 한 번에 섞어 평균을 내면 실제보다 높거나 낮게 잡힐 수 있습니다.

### 시즌 구간
- `1~2월`: 겨울 비수기
- `3~6월`: 1학기
- `7~8월`: 여름 비수기
- `9~12월`: 2학기

### 전체 결과
| 비교 항목 | 기존 기준 MAE | 개선 기준 MAE |
|---|---:|---:|
| 30분 후 평소값만 사용 | 31.1 | 25.8 |
| 60분 후 평소값만 사용 | 31.9 | 26.4 |
| 30분 후 최종 규칙 | 10.8 | 10.6 |
| 60분 후 최종 규칙 | 15.6 | 15.1 |
| 90분 후 최종 규칙 | 18.5 | 18.0 |

## 식당별 적용 방향

| 식당 | 60분 후 MAE 변화 | 이번 문서의 적용 방향 | 비고 |
|---|---:|---|---|
| 생활과학대 | `4.4 -> 3.2` | 시즌성 적용 | 개선 폭이 큼 |
| 제1학생회관 | `32.1 -> 28.5` | 시즌성 적용 | 개선 폭이 큼 |
| 제2학생회관 | `16.9 -> 15.7` | 시즌성 적용 | 개선 확인 |
| 상록회관 | `9.3 -> 9.5` | 시즌성 적용 | 손해 폭이 작아 통일안에 포함 |
| 제3학생회관 | `7.8 -> 10.5` | 시즌성 적용 | 손해 폭이 있어 모니터링 필요 |
| 학생생활관 | `27.5 -> 27.5` | 시즌성 미적용 | 기존 단순 규칙 유지 |

## 시간대별 적용 방향

| 예측 범위 | 먼저 보는 값 | 추가로 보는 값 | 적용 방향 |
|---|---|---|---|
| 30분 후 | 30분 뒤 평소값 | 지금이 평소보다 얼마나 많은지 또는 적은지 | 목표 시점 평소값을 기본으로 두고 현재 차이는 일부만 반영 |
| 60분 후 | 60분 뒤 평소값 | 현재 차이, 피크 이후 여부 | 평소값 쪽으로 더 강하게 되돌아가게 반영 |
| 90분 후 | 90분 뒤 평소값 | 현재 차이, 피크 이후 여부 | 더 먼 미래이므로 평소 흐름 복귀를 더 강하게 반영 |

## 60분 후 예측값 산출 예시

`60분 후 예측값 = 60분 뒤 평소값 + (현재값 - 현재 시점 평소값) × 현재 차이 반영률`

| 항목 | 예시 값 | 의미 |
|---|---:|---|
| 현재값 | 80명 | 지금 식당에 있는 인원 |
| 현재 시점 평소값 | 60명 | 같은 식당, 같은 요일, 같은 시간, 같은 시즌의 평균 |
| 60분 뒤 평소값 | 45명 | 예측하려는 시각의 과거 평균 |
| 현재 차이 | 20명 | 현재값 80명 - 현재 평소값 60명 |
| 현재 차이 반영률 | 0.4 | 현재의 특이 상황을 40%만 60분 뒤에 반영 |
| 최종 예측값 | 53명 | 45 + 20 × 0.4 |

## 운영 파라미터

| 파라미터 | 권장 초기값 | 설명 |
|---|---|---|
| 시즌 구간 | 1~2월, 3~6월, 7~8월, 9~12월 | 평소값 계산 시 과거 데이터를 나누는 기준 |
| 30분 현재 차이 반영률 | 크게 반영 | 현재 상황이 아직 이어질 가능성을 비교적 크게 봄 |
| 60분 현재 차이 반영률 | 중간 반영 | 현재 상황과 평소 흐름을 절충 |
| 90분 현재 차이 반영률 | 약하게 반영 또는 평소값 중심 | 시간이 멀어질수록 평소 흐름을 더 중요하게 봄 |
| 학생생활관 평균 기준 | 전 기간 평균 | 시즌성 이득이 크지 않아 단순 기준 유지 |

## 후보별 채택 여부

| 후보 | 채택 여부 | 이유 |
|---|---|---|
| 같은 요일 · 같은 시간대 평균 | 채택 | 가장 기본이 되는 평소값 |
| 같은 시즌 · 같은 요일 · 같은 시간대 평균 | 채택 | 30분·60분 후에서 평균값이 더 좋아짐 |
| 학생생활관 제외 시즌성 공통 적용 | 채택 | 운영 단순성과 설명 편의성을 우선한 기준 |
| 피크 이후 감소 허용 | 부분 채택 | 30분·60분 직접 감소는 과했으므로, 평소값 복귀 방식 안에서만 보조적으로 봄 |
| 같은 메인메뉴 기준 | 비채택 | 적용 가능한 구간이 약 7% 수준으로 적음 |
| 피크 이후 평균 감소량 직접 차감 | 비채택 | 30분 후 `10.8 -> 11.0`, 60분 후 `15.6 -> 16.3`으로 악화 |

HTML 보고서에서는 식당과 날짜를 선택해서 30분, 60분, 90분 그래프를 볼 수 있습니다. 범례 버튼은 여러 개를 동시에 선택해서 비교할 수 있습니다.

## DB 연동 프로젝트 제안

정적 HTML에 DB 접속 정보를 직접 넣는 방식은 권장하지 않습니다. 브라우저에 DB 주소와 계정이 노출되기 때문입니다. 실제 프로젝트에서는 백엔드가 DB를 조회하고, 프론트엔드는 백엔드 API만 호출하는 구조가 안전합니다.

| 구성 | 역할 | 초기 구현 범위 |
|---|---|---|
| DB | 실제 혼잡도, 식당, 메뉴, 시간대 데이터 저장 | 기존 MySQL 사용 |
| 백엔드 API | 평소값 계산, 30·60·90분 예측값 계산, 결과 반환 | `/api/congestion/predictions` 형태로 시작 |
| 프론트엔드 | 식당과 날짜 선택, 그래프 표시, 파라미터 표시 | 현재 HTML 화면을 API 기반 화면으로 전환 |
